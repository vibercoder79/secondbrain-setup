# 05 — Workflows: how you use the vault day to day

> TL;DR: Seven recurring workflows. Session start reads `Index.md`. Daily notes
> are the day's logbook. Project setup follows a 9-question onboarding. Deep
> research has two patterns. `/ingest` processes new notes. `/lint` checks the
> vault. Session end archives.

## Workflow 1: session start

When you start a new AI session you do **nothing manually** — the AI does it.
From the vault `CLAUDE.md`:

> ### On session start
> 1. Read `Index.md` at the vault root as the **vault cover** (regenerated by `/lint`).
> 2. Check `01 Inbox/` for new notes and show what's in there.

The AI reads `Index.md` on start — a pre-compiled overview of active projects,
areas, synthesis pages, recent daily notes, vault statistics. That saves
tokens and time compared to glob scans or full-text search. The pattern comes
from [Karpathy's LLM-Wiki](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f).

Example content of an `Index.md`:

```markdown
# SecondBrain Index

## Active projects (`02 Projekte/`)

- [[Example-Project - PMO HUB]] — `aktiv` | implementation | 2026-05-19
- [[My-Newsletter - PMO HUB]] — `aktiv` | concept | 2026-05-18
...

## Areas (`03 Bereiche/`)

- [[Skills]]
- [[Lifelong Learning]]
...

## Synthesis pages (`04 Ressourcen/`)

- [[Claude Code]] — synthesis of all Claude Code notes
- [[Gemini]]
...

## Vault statistics

- Notes total: 1247
- Active projects: 15
- Inbox count: 2
```

`Index.md` is regenerated by the `/lint` skill (step 6) — manual edits are lost.

## Workflow 2: daily notes — SSoT for daily activity

One file per day, named `YYYY-MM-DD.md`. The AI offers at session end to create
the entry or extend today's.

### Writing rules

1. **One file per day** — `05 Daily Notes/YYYY-MM-DD.md`
2. **Project sections** — one H2 section per project/topic with a tag:
   `## My-Project #my-project-tag`
3. **Tags in frontmatter** — one tag per touched project (plus `daily`)
4. **Wikilink to the hub** — `> [!info] Via Claude — [[My-Project - PMO HUB]]`
5. **Section content** — what was done, decisions (with wikilinks), open items

### The project hub pulls via Dataview

Every project hub has a `## Tagesaktivitaeten` section (the German name is part
of the convention so wikilinks keep working — the heading itself can be
anything you want):

````markdown
## Daily activity

```dataview
LIST file.link
FROM "05 Daily Notes"
WHERE contains(file.tags, "#my-project-tag")
SORT file.name DESC
LIMIT 30
```
````

The hub shows the last 30 days of activity automatically — you maintain the
daily note and the hub updates by itself. No duplicate maintenance.

### SSoT rules (what gets written WHERE)

| Data type | SSoT (write here) | Visible in |
| --------- | ----------------- | ---------- |
| Daily activity | `05 Daily Notes/YYYY-MM-DD.md` | project hub §Daily activity (Dataview) |
| Decisions | `02 Projekte/<project>/Decisions/ADR-XX.md` | hub §ADRs (index table) |
| Meetings | `02 Projekte/<project>/Meetings/YYYY-MM-DD.md` | hub §Meetings (Dataview or wikilinks) |
| Research | `02 Projekte/<project>/Research/*.md` | hub §Research (manual references) |

**Rule:** write in the SSoT, the hub only shows. If content needs to be in
multiple places → wikilink instead of copy-paste.

## Workflow 3: creating a project — 9-question onboarding

When you start a new project, you tell the AI:

> "Set up a project for Web Redesign."

The AI asks **10 questions** in a single block (all at once, not one by one):

```
0. Project language?          (a) German / (b) English
1. Project name?
2. One-sentence description?
3. Concrete goal?
4. Project type?              (Software / Consulting / Marketing / Personal / Other)
5. Stakeholder / customer?
6. Backlog tool?              (Linear / Teams Kanban / GitHub / none)
7. GitHub repo URL?           (optional)
8. Enable risk tracking?      (default: no)
9. Financials tracking?       (default: no)
```

After the answers:

- **Folder** is created: `02 Projekte/Web-Redesign/`
- **Required files:** `Web-Redesign - PMO HUB.md` + `Projekt-Governance.md`
- **Subfolders:** `Meetings/`, `Decisions/`, `Research/`, `assets/`
- **Opt-in:** `Risks/` (if answer 8 = yes), `Financials.md` (if answer 9 != none)
- **Backlog tool:** if Linear/Teams → MCP discovery, auto-create with confirmation;
  if GitHub → `gh` CLI; if none → "Pre-Backlog Action Items" section in the hub
- **Verification:** whitelist check (no `README.md` in the project root),
  wikilinks valid, Dataview syntax ok

The full methodology lives in `00 Kontext/Workflows/Projekt-Anlegen.md` —
AI-agnostic, readable by every AI. Claude Code automates it via `/projekt-init`;
Gemini and Codex walk through it manually (same questions, same result).

### Quick mode

If you say "default" or "no questions", the AI skips the onboarding and uses
the defaults from the methodology file. It only asks for the project name.

### Anatomy of a project hub

A typical `<project> - PMO HUB.md`:

```markdown
---
tags: [projekt]
status: aktiv
phase: concept
erstellt: 2026-05-21
aktualisiert: 2026-05-21
source: claude
chat_url: https://claude.ai/chat/...
governance: "[[Projekt-Governance]]"
---

# Web-Redesign

> One-sentence description of the project.

## Project goal
Concrete goal...

## Stack
(only for software projects)

## Daily activity
(Dataview query, filled automatically)

## Open decisions
(Dataview query on Decisions/ with status: open)

## Recent decisions
(Dataview query on Decisions/ with status: decided, LIMIT 5)

## Backlog
Link to the backlog tool or "Pre-Backlog Action Items"
```

The hub is the **landing page** of the project. Anyone who wants to understand
the project reads the hub.

## Workflow 4: filing deep research

When you do a thorough research run (Perplexity, ChatGPT Deep Research, Gemini
Long Context, etc.), there are two filing patterns.

### Simple research — one file

Directly as a single file in `04 Ressourcen/Research/`:

```
2026-05-21 AI market Europe comparison.md
```

Frontmatter with `source:`, `chat_url:`, `tags:`. Done.

### Complex research — a folder

When several related documents or graphics belong together:

```
2026-05-21 NIST CSF Tier 3 analysis/
├── README.md                  ← entry summary with wikilinks
├── 01-Document-Name.md        ← original document 1 (complete)
├── 02-Document-Name.md        ← original document 2 (complete)
├── 03-Synthesis.md            ← own synthesis
└── assets/
    └── graphic.svg
```

Rules:

- `README.md` is the entry point
- File all original artifacts **completely** — don't paraphrase
- Graphics into `assets/`, embedded via `![[graphic.svg]]`
- Sub-documents use `parent: "[[README]]"` in frontmatter
- File names with prefix numbers (`01-`, `02-`) to enforce reading order

**Trigger phrases** for the complex pattern:

- "archive that fully"
- "file this as a research bundle"
- "save the whole thing with all documents"

## Workflow 5: ingest — process and connect notes

After a research session or after creating an important note:

```
/ingest "Note XY"
```

What happens:

1. **Scan the vault** — what is thematically related?
2. **Set wikilinks** — bidirectional (in the new note AND in the existing ones)
3. **Update the synthesis page** — the `04 Ressourcen/<topic>/<topic>.md`
   landing file gets a new paragraph or section referring to the new note
4. **Update log.md** — chronology entry

This is the **compound effect** in action. Instead of knowledge sitting
isolated in single notes, the notes weave together. The synthesis page
(`04 Ressourcen/Claude Code/Claude Code.md`) gets richer with every new note —
it is Karpathy's "living wiki".

**Important:** ingest is **manual**, not a hook. You decide when it runs. A
hook on every edit would be too expensive (tokens) and too noisy (every trivial
edit would trigger).

Details: `skills/ingest/SKILL.md`.

## Workflow 6: lint — vault health check

Weekly (or on demand) check the whole vault:

```
/lint
```

Six steps:

1. **Find orphan notes** — notes without inbound wikilinks
2. **Suggest missing links** — topic matching, confirmed one by one
3. **Vault hygiene** — inbox count, frontmatter check, synthesis pages check
4. **Project compliance** — whitelist, required files, naming, broken wikilinks,
   decisions consistency, pre-backlog drift
5. **Create report** — `01 Inbox/YYYY-MM-DD Vault Lint Report.md`, log entry
6. **Regenerate Index.md** — vault cover (see workflow 1)

`/lint` **never changes anything automatically**. For each finding it asks
individually: "Should I correct this?"

Partial modes (when you only want to check one aspect):

- `/lint orphans` — only step 1
- `/lint inbox` — only the inbox part of step 3
- `/lint projekte` — only step 4
- `/lint index` — only step 6

Details: `skills/lint/SKILL.md`.

## Workflow 7: session end

When the AI notices a session is winding down or you end it:

1. **Create or extend the daily note** — section for today's project with
   "What was done", "Decisions", "Open"
2. **Save new insights** — if something valuable came out of the session that
   doesn't fit the daily note: a dedicated note in the appropriate folder
   (`02 Projekte/<project>/` or `04 Ressourcen/<topic>/`)
3. **Archive the chat history** (optional) — AI-specific:
   - Gemini: into `04 Ressourcen/Gemini/`
   - Codex: into `04 Ressourcen/Codex - OpenAI/`
   - Claude (Code/Desktop): usually not needed because Claude has its own history
4. **Clean up the inbox** if needed

## Putting it together: a typical day

```
Morning
└── Claude Code session
    ├── Index.md read, knows what's active
    ├── Inbox 2 notes → sorted
    ├── Work on project: ADR-04 decided, into Decisions/
    └── Session end: daily note entry for today

Noon
└── Perplexity research in the browser
    └── Web Clipper → 04 Ressourcen/Research/2026-05-21 Topic X.md

Afternoon
└── Claude Code session
    ├── /ingest "2026-05-21 Topic X" → wires it into existing knowledge
    ├── Daily note extended
    └── Session end

Friday
└── /lint
    ├── 3 orphans found, 2 linked
    ├── 1 project with compliance drift fixed
    └── Index.md regenerated
```
