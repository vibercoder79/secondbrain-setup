# 07 — Customizing: your own paths, your own tools, migration

> TL;DR: Change the vault path, add your own workflows, choose a language,
> migrate from an existing PARA vault, build your own skills. What you can
> touch and what you can't.

## What you can touch

Everything in your own vault is yours. Three places are especially relevant:

### 1. The vault path

The default in the templates is `~/Obsidian/SecondBrain/`. If you want a
different path (e.g. `~/Vaults/Brain/` or `~/Documents/Knowledge/`), change it:

**In the global configs:**

- `~/.claude/CLAUDE.md` — path in the "Second Brain" section
- `~/.gemini/GEMINI.md` — `@-import` path
- `~/.codex/AGENTS.md` — reference path
- `~/Library/Application Support/Claude/claude_desktop_config.json` —
  Filesystem MCP argument

**In the vault `CLAUDE.md` itself:**

If you have absolute paths in the vault `CLAUDE.md` (example paths in workflows
etc.), update them. The templates in this repo use tilde notation
(`~/Obsidian/SecondBrain/`) — if you stay with tilde, you usually only need
to update the global configs.

### 2. Personal profile

The files in `00 Kontext/` are your content:

- `Über mich.md` — who you are, professional background, focus
- `ICP.md` — if you have customers: ideal target group
- `Angebot.md` — what you offer
- `Schreibstil.md` — tone, style rules (AIs use it to write in your voice)
- `Branding.md` — brand values, colors, design principles

These files are **not included** in the repo — they are individual. Write them
yourself. Tip: start minimal (3-4 sentences per file), extend later.

### 3. The workflow file

`00 Kontext/Workflows/Projekt-Anlegen.md` is included in the template and you
can adapt it:

- Different onboarding questions
- Different defaults per project type
- Different required folders (e.g. if you always want a `Documents/` folder)

As long as the pattern stays AI-agnostic (all AIs read the same file and run
the same workflow), you can change it freely.

**If you add your own workflows** (e.g. `Meeting-Vorbereiten.md`,
`Newsletter-Schreiben.md`), create a new file in `00 Kontext/Workflows/`. The
vault `CLAUDE.md` (root) references it from the matching section — example:

```markdown
## Writing the newsletter

Methodology: [[00 Kontext/Workflows/Newsletter-Schreiben]]
```

Every AI then sees the new workflow.

## What you should not touch

### `Index.md`

Regenerated by `/lint`. Manual edits are lost. The source of truth is the
individual notes.

### `log.md`

Append-only. Processing chronology from `/ingest` and `/lint`. Don't edit
existing entries. If something was logged incorrectly, write a correction
entry below it, not on top.

### Mandatory frontmatter fields

`source:`, `chat_url:`, `tags:` — these are convention. If you omit them, the
skills lose the ability to manage the notes (`/lint` warns, Dataview queries
return nothing).

## Language: German or English?

The setup runs in German (Tobias' default). The templates have English
variants:

- `Projekt-Struktur.md` (DE) + `Projekt-Struktur.en.md` (EN)
- `User-Story-Template.md` (DE) + `User-Story-Template.en.md` (EN)

If you want to use the setup primarily in English:

1. Activate EN templates everywhere
2. In `Workflows/Projekt-Anlegen.md` set the default language to `en`
3. Change language hints in the global configs to English
4. Vault `CLAUDE.md`: optionally translate fully to English (it's not
   mandatory — multilingual content is fine)

The setup has a built-in **language logic** in the project-creation workflow:
question 0 is the language choice and the AI then generates DE or EN content
per project. File and folder names stay the same (whitelist), only the content
is translated.

## Migrating from an existing PARA vault

If you already have an Obsidian vault (or a Notion/Logseq) that is roughly
PARA-shaped:

### Step 1: back up

```bash
cp -r ~/Obsidian/MyVault ~/Obsidian/MyVault.backup
```

### Step 2: add the required folders

```bash
cd ~/Obsidian/MyVault
mkdir -p "00 Kontext" "01 Inbox" "02 Projekte" "03 Bereiche" "04 Ressourcen" "05 Daily Notes" "06 Archiv" "07 Anhaenge"
```

If you had different folder names ("Projects" instead of "02 Projekte"), you
can rename them OR adjust the templates (in the vault `CLAUDE.md` and in
`Projekt-Anlegen.md`).

### Step 3: create the `CLAUDE.md`

Copy `templates/vault/CLAUDE.md` to `~/Obsidian/MyVault/CLAUDE.md`. Adjust
paths if your folders are named differently.

### Step 4: fill `00 Kontext`

Create the profile files (even if rough).

Copy `templates/vault/00 Kontext/Projekt-Struktur.md` and
`Workflows/Projekt-Anlegen.md`.

### Step 5: align existing projects

For each existing project:

- Make sure it has a folder (no loose single file)
- Rename the hub file to `<project> - PMO HUB.md`
- Add `Projekt-Governance.md` (even minimal: `backlog_tool: none`)
- Create subfolders `Meetings/`, `Decisions/`, `Research/`, `assets/`

If you have 20+ projects: not all at once. Step by step.

### Step 6: connect the AIs

See chapter 04.

### Step 7: `/lint`

If you have Claude Code and the lint skill installed: run it. It will show you
every compliance violation you still need to fix.

## Other backlog tools

Default support in `/projekt-init`: Linear, M365 Teams Kanban, GitHub Issues,
none.

### Notion

Not currently MCP-integrated in the default setup. Workaround:

- Answer question 6 with `none`
- The hub gets a "Pre-Backlog Action Items" section
- You push items manually into the Notion board

If you want Notion MCP integration: community MCPs exist (e.g.
[mcp-notion](https://github.com/danhilse/notion_mcp)), and you can add your
own discovery reference to the skill.

### Asana, ClickUp, Jira

Same pattern: no default support, but when an MCP exists the hybrid strategy
can be implemented analogously (see the skill reference
`backlog-discovery-linear.md` as a template).

## Building your own skills

If you want to extend the vault pattern — e.g. a `/newsletter-init` skill or
`/cv-update` skill:

1. Document the methodology **first in `00 Kontext/Workflows/`** (AI-agnostic)
2. Build the Claude skill in `~/.claude/skills/<name>/` to automate it
3. Add MCP discovery patterns if needed
4. Documentation in the vault under `03 Bereiche/Skills/<name>.md`

Detailed lifecycle workflow in the `skill-creator` skill from Tobias' main
repo: [vibercoder79/claudecodeskills](https://github.com/vibercoder79/claudecodeskills)
(skill `skill-creator`).

## Versioning your vault

Recommended: put the vault itself under Git. Not for backup (Obsidian Sync,
iCloud, OneDrive are for that), but for history.

```bash
cd ~/Obsidian/SecondBrain
git init
echo ".obsidian/workspace.json" >> .gitignore  # don't version the workspace state
echo "07 Anhaenge/" >> .gitignore               # optional: don't version images/PDFs
git add CLAUDE.md AGENTS.md Index.md log.md "00 Kontext" "02 Projekte" "03 Bereiche" "04 Ressourcen" "05 Daily Notes" "06 Archiv"
git commit -m "Initial Vault"
```

You can commit per daily-note session. Or once a week. That's a matter of
taste.

**Important:** if your vault contains secrets (it shouldn't), check before
every commit. Generic vault contents are usually non-sensitive.

## Performance with large vaults

Tobias' vault has more than 1200 notes. Up to that point everything runs
smoothly. Beyond ~3000-5000 notes you may run into trouble:

- `/lint` slows down (glob across all files)
- `Index.md` might get too long
- AI token budget gets tight on session start

Mitigations:

- Use `06 Archiv/` more aggressively
- Filter `Index.md` (only `aktiv` status projects)
- External search tools like [qmd](https://github.com/karpathy/llm.c) (BM25 /
  vector search over local Markdown) as a complement

## When you leave the setup

The nice thing about this setup is: **you have nothing to lose**. It's all
Markdown in folders. If you don't want Obsidian tomorrow, you open it with any
text editor. If you don't want AIs anymore, you delete the global configs —
the vault stays.

The only hard dependency is Obsidian's **Dataview plugin** for the hub
queries. If you lose Dataview, the hubs only show the raw code block instead
of the evaluated results. But the data (daily notes, ADRs, etc.) remains
readable and searchable.
