# 01 — Philosophy: Why this setup exists

> TL;DR: A single Markdown vault as the shared brain for all your AIs. PARA gives
> the order, Karpathy's LLM-Wiki pattern brings it to life, wikilinks weave it
> together. Three or four AIs write into the same vault without stepping on each
> other.

## The problem

AIs are useful, but they are islands. You research in Perplexity, plan in Claude
Desktop, code in Claude Code, draft emails with Gemini, generate code skeletons
with Codex. Each of these AIs has its own history, its own context, its own
memories. If you talk to Claude on Monday about a project and discuss the same
project with Gemini on Tuesday, Gemini starts from zero.

Three symptoms are typical:

1. **Scattered knowledge.** A research result from Perplexity, a decision from a
   Claude chat, a code sketch from Codex. Three places, three formats, no shared
   index.
2. **No compound effect.** Every session begins with context-rebuilding. You
   repeatedly explain what the project is, where it stands, what has already been
   decided.
3. **No ownership of your own knowledge.** If the chat history is gone (deleted,
   not exported, AI vendor switch), the insight is gone with it.

Classic remedies like "memory" or "custom instructions" inside a single AI do
not solve this — they center the knowledge in the AI, not with the user.

## The two inspirations

### Tiago Forte — PARA and Building a Second Brain

PARA is a four-folder method: **P**rojects, **A**reas, **R**esources, **A**rchives.
Plus an **Inbox** for the unsorted. The idea: every piece of information has a
clear home, sorted by **actionability**, not by topic. Projects have a goal and
an end date, Areas are ongoing responsibilities without an end date, Resources
are reference material, Archives are completed.

PARA gives this setup its **skeleton**: a predictable folder structure in which
AIs and humans know where something belongs.

### Andrej Karpathy — LLM-Wiki pattern

In his gist `LLM-as-a-Wiki` ([source](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f))
Karpathy described a different pattern: instead of retrieving individual documents
via RAG on demand, an LLM maintains a **structured Markdown wiki incrementally**.
New sources are incorporated, cross-references updated, contradictions flagged.
Three operations are central:

- **Ingest:** process a new source, update 10-15 wiki pages simultaneously, set
  wikilinks.
- **Query:** search the wiki, synthesize answers with citations, turn valuable
  insights into new pages.
- **Lint:** periodic health checks — contradictions, stale claims, orphan pages,
  knowledge gaps.

Karpathy's idea gives this setup its **liveliness**: the vault is not just
stored, it is actively developed. Wikilinks connect, synthesis pages condense,
lint reports surface drift.

## The synthesis

PARA without Karpathy: a clean filing system, but static. Karpathy without PARA:
living connections, but no one knows where anything belongs. The combination is
more than the sum of its parts:

| Aspect        | PARA                       | Karpathy LLM-Wiki              | This setup |
| ------------- | -------------------------- | ------------------------------ | ---------- |
| Approach      | Organizing system          | Processing system              | Both       |
| Organization  | Fixed folder structure     | Schema-based, LLM decides      | Fixed folders + LLM workflows |
| Who maintains | User files things away     | LLM curates actively           | User files, LLM condenses |
| Strength      | Structure, findability     | Synthesis, compound effect     | Both       |
| Weakness      | Knowledge often isolated   | No filing structure            | — |

## Multi-AI: one vault, many gates

The second lever is **AI agnosticism**. The vault is the single source of truth,
not the chat history of any specific AI. Each AI gets a thin config file that
points back to the vault:

- `~/.claude/CLAUDE.md` → points to `~/Obsidian/SecondBrain/CLAUDE.md`
- `~/.gemini/GEMINI.md` → points to `~/Obsidian/SecondBrain/CLAUDE.md`
- `~/.codex/AGENTS.md`  → points to `~/Obsidian/SecondBrain/CLAUDE.md`
- Claude Desktop → reads the vault via a Filesystem MCP server

That is the **hub-and-spoke pattern** (chapter 02). Change a rule in the vault
and every AI sees it immediately. Switch AI providers and the knowledge stays.

## What you get out of it

When the setup runs, you gain three things:

1. **Continuity between sessions.** On session start the AI reads `Index.md`
   (the vault cover, generated by the `/lint` skill) and instantly knows which
   projects are active, where things stood last week, where decisions are
   documented.
2. **Compound effect.** Every new research note is wired into the existing
   knowledge via `/ingest`. Wikilinks become bidirectional. Synthesis pages
   grow. The vault gets better with every source.
3. **Portability.** When a better model or a different vendor shows up tomorrow,
   you move the config file. The knowledge stays, because it is yours and it
   lives in Markdown, not in some vendor database.

## What you have to bring

Three things are non-negotiable:

1. **Discipline when saving.** One daily note per day, new notes with
   frontmatter, regularly empty the inbox. The system is only as good as the
   inputs.
2. **Trust in Markdown.** No proprietary formats, no databases. Markdown +
   wikilinks + YAML frontmatter. If that feels too old-school for you, the
   setup isn't for you.
3. **A text editor with wikilink support.** Recommendation: [Obsidian](https://obsidian.md).
   Free, local, plugin-friendly. You can also use others (Logseq, Foam in VS
   Code, plain Markdown), but this setup is built around Obsidian conventions
   (wikilinks `[[Name]]`, Dataview queries, Excalidraw plugin).

## Who doesn't need this

- Anyone who only uses a single AI and is happy with that.
- Anyone without recurring projects and no need for compound insights.
- Anyone for whom in-app "memory" is enough.

But if you seriously use two or more AIs day to day and notice that your
knowledge is scattered — read on.
